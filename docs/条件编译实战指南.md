## 核心设计思路

MicroPython 设备资源极其有限（Flash 通常只有 1–16 MB，RAM 甚至低至 128 KB），无法像 CPython 那样在运行时动态加载。条件编译让你在**刷入前**就决定哪些代码进入设备，避免运行时条件判断的额外开销和死代码占据空间。

```
┌──────────────┐    预处理（libcst）    ┌──────────────────┐    编译（mpy-cross）    ┌─────────────┐
│  源码（含宏）  │ ──────────────────→  │  展开为 if/True   │ ──────────────────→  │  精简字节码   │
│              │   if False 块标记    │  /False 的源码     │   死代码消除         │             │
└──────────────┘                      └──────────────────┘                       └─────────────┘
      ↑                                                                                ↓
  with target("ESP32"):                                                        只保留匹配的代码
  @feature("ble")
```

`target` 和 `feature` 在语法上完全一致，语义区别仅在于：

| | target | feature |
|--|--------|---------|
| 语义 | 硬件平台（互斥） | 功能特性（可组合） |
| 来源 | 设备自动识别 / `--target` | board_tags 隐式激活 / `--feature` 追加 |
| 典型值 | `ESP32`, `RP2040`, `STM32` | `wifi`, `ble`, `sdcard`, `debug` |

## 实战场景一：单源码支持多款开发板

### 问题

你维护一个 IoT 项目，需要同时支持 ESP32-S3 和 RP2040 两款开发板。它们的引脚定义、外设驱动、Flash 布局都不同。传统做法是维护两份代码（`main_esp32.py` / `main_rp2040.py`），重复代码难以同步。

### 方案

用一个 `main.py`，用 `target()` 区分平台差异：

```python
# main.py
import time

# ── 引脚定义 ──
with target("ESP32"):
    from machine import Pin
    LED_PIN = 2
    I2C_SCL = 22
    I2C_SDA = 21

with target("RP2040"):
    from machine import Pin
    LED_PIN = 25   # RP2040 板载 LED 在 GPIO 25
    I2C_SCL = 5
    I2C_SDA = 4

# ── 初始化（公共逻辑）──
led = Pin(LED_PIN, Pin.OUT)
i2c = machine.I2C(0, scl=Pin(I2C_SCL), sda=Pin(I2C_SDA))

# ── 平台特有驱动 ──
with target("ESP32"):
    import esp32
    def deep_sleep(sec):
        esp32.wake_on_ext0(Pin(14, Pin.IN), 1)
        machine.deepsleep(sec)

with target("RP2040"):
    def deep_sleep(sec):
        # RP2040 不支持 esp32 式的 deepsleep，用 lightsleep 替代
        machine.lightsleep(sec * 1000)

# ── 主循环 ──
while True:
    led.toggle()
    time.sleep(1)
```

### 刷入

```bash
# 连上 ESP32 时，工具自动识别 target
pyrcli flash COM3 main.py /main.py

# 离线指定
pyrcli flash COM3 main.py /main.py --target ESP32
```

### 使用装饰器实现多平台同函数名

当不同平台需要同名的入口函数时，用 `@target` 装饰器：

```python
@target("ESP32")
def init_sensors():
    import dht
    return dht.DHT22(Pin(15))

@target("RP2040")
def init_sensors():
    # RP2040 用不同的温度传感器
    from machine import ADC
    return ADC(4)   # 内部温度传感器

sensor = init_sensors()   # 只保留匹配平台的实现
```

工具会自动分析：两个实现中与当前 active_tags 匹配的保留原函数定义，不匹配的包裹在 `if False:` 中。

## 实战场景二：按功能特性裁剪固件

### 问题

你的项目有多项可选功能：WiFi 联网、蓝牙配置、SD 卡日志、OLED 显示。不同客户需要不同组合，如果全部编译进去，固件体积会超标。

### 方案

用 `feature()` 将每个可选功能包在 guard 内：

```python
# config.py
SSID = "my_iot"
PASSWORD = "secret"

# ── WiFi 功能 ──
with feature("wifi"):
    import network
    import ujson

    def upload_data(data):
        wlan = network.WLAN(network.STA_IF)
        wlan.active(True)
        wlan.connect(SSID, PASSWORD)
        # ... HTTP POST ...

# ── 蓝牙配置功能 ──
with feature("ble"):
    import bluetooth

    def ble_config():
        # ... BLE 服务 ...
        pass

# ── SD 卡日志功能 ──
with feature("sdcard"):
    import os

    def log_to_sdcard(msg):
        with open("/sd/log.txt", "a") as f:
            f.write(msg + "\n")

# ── 主逻辑 ──
with feature("wifi"):
    upload_data({"temp": 25.5})

with feature("sdcard"):
    log_to_sdcard("boot complete")
```

### 刷入时选择组合

```bash
# 仅 WiFi 版本（ESP32 默认已激活 wifi）
pyrcli flash-program COM3 src/ --target ESP32

# WiFi + SD 卡版本
pyrcli flash-program COM3 src/ --target ESP32 --feature sdcard

# 全功能版本
pyrcli flash-program COM3 src/ --target ESP32 --feature sdcard,ble,oled

# 禁用某个特性
pyrcli flash-program COM3 src/ --target ESP32 --no-feature wifi
```

### board_tags 自动激活

在 `pyproject.toml` 中定义 board 与 feature 的关联：

```toml
[tool.pyrite.board_tags]
# 当 target=ESP32 时，自动激活 wifi
# 当 target=ESP32_S3 时，自动激活 wifi 和 ble
ESP32_S3 = ["ESP32_S3", "wifi", "ble"]
ESP32_C3 = ["ESP32_C3", "wifi", "ble"]
```

这样 `--target ESP32_S3` 时，`wifi` 和 `ble` 自动激活，无需手动 `--feature`。

## 实战场景三：开发版 vs 生产版差异管理

### 问题

开发过程中需要调试日志、模拟数据、交互式测试；生产版本需要去掉这些，还要加入看门狗、错误上报等生产特性。手动注释非常容易遗漏。

### 方案

用 `feature("debug")` 区分：

```python
# utils.py
with feature("debug"):
    def log(msg):
        print(f"[DEBUG] {msg}")

    def simulate_sensor():
        return {"temp": 25.0, "hum": 60.0}

with feature("prod"):
    import machine
    WDT_TIMEOUT = 30000  # 30 秒看门狗

    def init_watchdog():
        wdt = machine.WDT(timeout=WDT_TIMEOUT)
        return wdt

# 代码中可以安全地调用这些函数，
# 不匹配的 target/feature 会被预处理消除
```

### 工作流

```bash
# 开发时：打开 debug
pyrcli flash-program COM3 src/ --feature debug

# 发布时：开启生产特性，关闭调试
pyrcli flash-program COM3 src/ --no-feature debug --feature prod
```

### 用函数装饰器实现依赖注入

```python
# 开发版传感器（模拟数据）
@feature("debug")
def get_temperature():
    import random
    return round(20 + random.random() * 10, 1)

# 生产版传感器（真实硬件）
@feature("prod")
def get_temperature():
    from machine import ADC
    adc = ADC(4)
    voltage = adc.read_u16() * 3.3 / 65535
    return round(27 - (voltage - 0.706) / 0.001721, 1)

# 调用点无需关心版本差异
temp = get_temperature()
```

## 常用设计模式

### 模式一：平台抽象层 (PAL)

将平台相关代码收拢到一个文件中统一管理：

```python
# pal.py — Platform Abstraction Layer
import machine

# ── 睡眠模式 ──
@target("ESP32")
def sleep_ms(ms):
    machine.sleep(ms)

@target("RP2040")
def sleep_ms(ms):
    # RP2040 没有 machine.sleep()
    import time
    time.sleep_ms(ms)

# ── 唯一 ID ──
@target("ESP32")
def device_id():
    import esp32
    return ''.join(f"{b:02x}" for b in esp32.mac())

@target("RP2040")
def device_id():
    from machine import unique_id
    return ''.join(f"{b:02x}" for b in unique_id())
```

业务代码 `main.py` 只调用 `pal.sleep_ms(100)`，完全不关心底层差异。

### 模式二：功能模块开关

将整块功能独立成文件，用 manifest.py 的 `features` 过滤决定是否刷入：

```python
# manifest.py
module("main.py")
module("pal.py")

# 以下文件仅在对应 feature 激活时刷入
module("wifi_manager.py", features=["wifi"])
module("ble_service.py", features=["ble"])
module("sdcard_logger.py", features=["sdcard"])
package("drivers/", features=["sdcard"])
```

这样不仅代码级消除，连文件级都消除了——不匹配的功能根本不会被刷入设备。

### 模式三：分层 feature 设计

用 feature 的叠加实现精细控制粒度：

```python
# ── 日志级别分级 feature ──
with feature("log_minimal"):
    # 仅错误日志
    def log_error(msg): print(f"[E] {msg}")

with feature("log_verbose"):
    def log_error(msg):   print(f"[E] {msg}")
    def log_info(msg):    print(f"[I] {msg}")
    def log_debug(msg):   print(f"[D] {msg}")

# 刷入时选择日志级别
# --feature log_verbose   → 详细日志
# --feature log_minimal   → 仅错误日志
# 两个都不加          → 无日志（最省空间）
```

## manifest.py 联合过滤

manifest.py 中的 `features` 与源码中的 `feature()` 块共同作用，形成两层过滤：

| 层次 | 机制 | 效果 |
|------|------|------|
| 文件级 | manifest.py `features` 参数 | 不匹配的文件根本不刷入设备 |
| 代码级 | 源码中 `with feature()` / `@feature()` | 刷入的文件内，不匹配的代码块被消除 |

```python
# manifest.py
module("sensor_drivers.py", features=["sensor"])

# sensor_drivers.py
with feature("sensor_temperature"):
    class TemperatureSensor:
        ...

with feature("sensor_humidity"):
    class HumiditySensor:
        ...
```

刷入 `--feature sensor,sensor_temperature` 时：
- `sensor_drivers.py` 被刷入（因 `sensor` 匹配文件级过滤）
- 文件中仅 `TemperatureSensor` 被保留（代码级过滤）
- `HumiditySensor` 被消除

## 调试与排查

### 预处理结果预览

在命令行快速查看某个文件预处理后的效果，无需连设备：

```powershell
python -c "
from cli.utils.preprocessor import preprocess
src = open('main.py').read()
result = preprocess(src, {'ESP32', 'wifi'}, 'main.py')
print(result)
"
```

### 确认 active_tags

`pyrite-cli` 在有设备连接时自动输出识别的 target 和 features：

```powershell
pyrcli board-info COM3
# 输出中会显示识别的平台信息
```

项目的 `.pyrite_hash.json` 中也记录每次刷入时的 active_tags 快照（需配合 project flash 使用）。

### 查看哪些代码被保留

刷入后通过 `pyrcli run` 验证函数是否存在：

```powershell
pyrcli run COM3 "import sensor; print(dir(sensor))"
```

如果某个函数未定义，检查：
1. 该函数是否被 `feature/target` guard 包裹
2. 当前 active_tags 是否包含所需 tag
3. 是否有静态警告提示"所有实现均不匹配"

## 常见陷阱

### 陷阱 1：`@target` 装饰的函数调用点未被 guard

```python
@feature("wifi")
def connect_wifi():
    ...

# 连接 WiFi
connect_wifi()           # ← 警告：裸调用
```

如果 `wifi` 未激活，`connect_wifi` 被消除，但调用点还在，运行时 `NameError`。

**解决**：将调用点也放入 guard：

```python
with feature("wifi"):
    connect_wifi()
```

工具会静态分析并输出黄色警告，留意即可。

### 陷阱 2：`with` 内嵌 return/break/continue

`with target(...)` 预处理后变成 `if True/False:`，MicroPython 中 `if` 块内 `return` / `break` / `continue` 合法。但在某些位置（如函数体最外层用 `with target` 替代 `if` 时），需要注意语义不变。

### 陷阱 3：跨文件引用未 guard 的函数

```python
# utils.py
@feature("wifi")
def scan(): ...

# main.py
from utils import scan

# 这里没有 guard，如果 wifi 未激活，scan 在 utils.py 中被消除
from utils import scan   # ← ImportError 或 NameError
```

工具**不会**跨文件检测。建议在 `main.py` 中也用 `with feature("wifi")` 包裹调用。

### 陷阱 4：字符串字面量中的路径

tag 匹配是**精确字符串匹配**。注意：
- target 会自动转大写：`--target esp32` → `ESP32`
- feature 保持原样：`--feature SdCard` 需要代码中用 `feature("SdCard")`

### 陷阱 5：文件级过滤和代码级过滤不一致

manifest.py 中用 `features=["wifi"]` 过滤了文件，但文件内部还有 `with feature("ble")`。如果只开了 `wifi` 但没开 `ble`，文件虽然刷入了，但其中的 ble 代码块被消除——这通常是预期行为。反过来说，如果 manifest 没过滤但代码内部全部被消除，文件刷入了一个空文件，浪费设备空间。建议要么在 manifest 层过滤，要么在代码层 guard，不要双重依赖。

## 总结

| 场景 | 推荐工具 | 效果 |
|------|----------|------|
| 多平台同源 | `@target("...")` 装饰器 | 同名函数按平台保留实现 |
| 可选功能裁剪 | `with feature("..."):` 块 | 按 tag 保留或消除代码块 |
| 开发/生产差异 | `feature("debug")` / `feature("prod")` | 一行命令切换构建模式 |
| 文件级裁剪 | manifest.py `features` 参数 | 不匹配的文件不刷入 |
| 平台抽象层 | `@target` + `with target` 合并到 PAL 文件 | 业务代码零平台耦合 |

条件编译的核心价值在于：**一份源码，多套构建，按需组合**。结合 pyrite-cli 的 project 命令（`project flash` / `project status` / `project pull`）和哈希增量刷入，可以在同一项目目录下管理不同硬件平台和功能配置的固件构建。
