## 传输层 (`utils/transport.py`, `serial_transport.py`, `webrepl_transport.py`)

### 设计目标

将底层通信方式从「硬编码的 pyserial」解耦为 **Transport 抽象接口**，使得 `Flash.py` 核心逻辑无需关心数据是通过串口还是 WebSocket 传输。

### Transport ABC (`transport.py`)

抽象基类，定义了所有传输实现必须实现的接口，并提供了通用的接收缓冲逻辑：

| 方法/属性 | 类型 | 说明 |
|-----------|------|------|
| `connect()` | abstract | 建立连接（串口打开 / WebSocket 握手） |
| `disconnect()` | concrete | 断开连接，清空接收缓冲 |
| `write(data)` | concrete | 写入数据，委托 `_raw_write()` |
| `read(size)` | concrete | 读取数据，优先消费内部 `_rx_buf`，不足时自动 `_fill_buf()` |
| `in_waiting` | property | 接收缓冲区可用字节数 |
| `reset_input_buffer()` | concrete | 清空收发缓冲区 |
| `reset_output_buffer()` | concrete | 清空发送缓冲区（仅串口有意义） |
| `is_connected` | abstract | 是否已连接 |
| `_raw_write(data)` | abstract | 子类实现的具体写入 |
| `_raw_read(size)` | abstract | 子类实现的具体读取 |
| `_raw_in_waiting()` | abstract | 子类实现的可用字节数查询 |
| `_fill_buf()` | concrete | 从底层读取数据填充内部缓冲区 |

**接收缓冲机制**：`Transport` 内部维护 `_rx_buf`，`read()` 和 `in_waiting` 都会先尝试 `_fill_buf()` 从底层拉取更多数据再进行消费。这避免了在每处调用点都编写 `if in_waiting: read()` 的重复代码。

### SerialTransport (`serial_transport.py`)

基于 `pyserial` 实现，是串口通信的封装：

```
SerialTransport
├── __init__(port, baudrate, timeout)
├── connect()         → serial.Serial(port, baudrate, timeout, write_timeout)
├── disconnect()      → ser.close()
├── _raw_write(data)  → ser.write(data)
├── _raw_read(size)   → ser.read(size)
├── _raw_in_waiting() → ser.in_waiting
├── reset_output_buffer() → ser.reset_output_buffer()
└── is_connected      → ser is not None and ser.is_open
```

- `connect()` 在打开串口后自动调用 `reset_input_buffer()` 清除启动残留数据。
- 断开连接时同步清理父类 `_rx_buf`。

### WebREPLTransport (`webrepl_transport.py`)

基于 `websocket-client` 实现 WebREPL 协议，允许通过 WiFi 连接 MicroPython 设备：

```
WebREPLTransport
├── __init__(url, password)
│   ├── url:      ws://device_ip:8266/
│   └── password: CLI 参数 / 环境变量 / 交互输入
├── connect()
│   ├── 1. 创建 WebSocket 连接
│   ├── 2. 接收 JSON 挑战 {"uid":"...", "nb":9}
│   ├── 3. 计算 SHA256(password + uid) → 取前 nb 位 Hex
│   └── 4. 发送认证响应，验证返回的 ":" 前缀
├── disconnect()     → ws.close()
├── _raw_write(data) → ws.send(data, OPCODE_BINARY)
├── _raw_read(size)  → 返回空（由 _fill_buf 驱动）
├── _fill_buf()      → ws.recv() 填充 _rx_buf
└── is_connected     → ws is not None
```

**密码解析链**：`CLI --password` 参数 → `PYRITE_WEBREPL_PASSWORD` 环境变量 → `getpass()` 交互输入。

## 类型定义 (`utils/types.py`)

### PyriteConfig 数据类

替代了原来 `Flash.py` 中通过裸 `dict` 访问配置的方式，提供类型安全和 IDE 自动补全：

```python
@dataclass
class PyriteConfig:
    chunk_size: int = 4096          # 每次写入块大小
    download_threads: int = 4       # 存根下载并发数（1-12）
    auto_compile: bool = True       # 是否自动编译 .py → .mpy
    verify: str = "size"            # 校验模式: off / size / crc32
    max_retries: int = 2            # 校验失败重试次数
    board_tags: Dict[str, List[str]]  # 板级标签映射（从 pyproject.toml 补充）
```

使用方式从 `self.config["chunk_size"]` 变为 `self.config.chunk_size`。

## 配置系统 (`utils/config.py`)

负责从文件系统加载配置，合并多个配置源：

| 函数 | 返回值 | 说明 |
|------|--------|------|
| `_load_config()` | `PyriteConfig` | 从 CWD 向上搜索 `.pyrite_config.json`，合并 `pyproject.toml` 的 board_tags |
| `create_default_config()` | `str` | 在工作目录创建默认 `.pyrite_config.json` |

**配置加载流程**：
1. 从 `PyriteConfig()` 默认值开始
2. 逐级向上查找 `.pyrite_config.json`，找到后合并 JSON 中的字段
3. 逐级向上查找 `pyproject.toml`，合并 `[tool.pyrite.board_tags]`
4. 覆盖顺序：内置默认 < JSON 文件 < pyproject.toml board_tags

## 编译模块 (`utils/compiler.py`)

将 `mpy-cross` 命令行调用封装为 Python API，支持串行和并行两种模式：

| 函数 | 说明 |
|------|------|
| `_compile_to_mpy(local_path, bytecode_ver, arch)` | 编译单个 .py → .mpy，失败回退到原始 .py |
| `_compile_files_parallel(local_paths, bytecode_ver, arch, max_workers)` | 使用 `ThreadPoolExecutor` 并行编译多个文件 |

- 使用 `mpy_cross` Python API，不再依赖外部 `mpy-cross` 可执行文件
- 编译失败时打印黄色警告并静默回退到 `.py`，不影响刷入流程
- 临时文件存放在 `tempfile.mkdtemp()` 创建的临时目录，退出时自动清理

## CLI 入口 (`main.py`) 变更

### 传输层选择

所有需要设备通信的命令新增两个全局选项：

| 选项 | 类型 | 默认 | 说明 |
|------|------|------|------|
| `--ws` | `str` | `None` | WebREPL WebSocket URL，如 `ws://192.168.4.1:8266/` |
| `--password` | `str` | `None` | WebREPL 密码（省略时从环境变量或交互输入） |

### _mp_factory() 工厂函数

统一管理 `MicroPython` 实例的创建和传输层选择：

```python
def _mp_factory(port, baudrate, timeout, ws, password) -> MicroPython:
    if ws:
        return WebREPLMicroPython(url=ws, password=password, timeout=timeout)
    return MicroPython(port=port, baudrate=baudrate, timeout=timeout)
```

`WebREPLMicroPython` 是 `MicroPython` 的子类，内部创建 `WebREPLTransport` —— `Flash.py` 完全不感知 WebREPL 的存在。
不提供 `--ws` 时自动使用串口，行为与之前完全一致。

## Flash.py 核心重构

### 状态机简化

| 移除项 | 替代方案 |
|--------|----------|
| `_in_raw` 标记 | 每次 `_enter_raw_repl()` 执行完整初始化 |
| `_kbd_set` 标记 | 同上 |
| `_setup_kbd_intr()` | 集成到 `_init_device_state()` |
| `_restore_kbd_intr()` | 不再需要，连接释放时自动恢复 |
| 各方法中的 `if not is_connected: connect()` 散落逻辑 | 统一由 `_ensure_connected()` 处理 |

### 新增核心方法

- `_ensure_connected(retries=3)` — 检查连接状态，未连接时自动重试重连
- `_init_device_state()` — 完整的设备初始化序列：
  1. 多次 Ctrl+C 中断可能卡住的程序
  2. `kbd_intr(-1)` 禁用 Ctrl+C 触发软重启
  3. 进入原始 REPL 模式（Ctrl+A）
  4. 检测设备 mpy 版本和架构

### 传输层适配

`Flash.py` 中所有 `self.ser.*` 操作迁移为 `self.transport.*`：

| 原写法 | 新写法 |
|--------|--------|
| `self.ser.write(data)` | `self.transport.write(data)` |
| `self.ser.read(n)` | `self.transport.read(n)` |
| `self.ser.in_waiting` | `self.transport.in_waiting` |
| `self.ser.is_open` | `self.transport.is_connected` |
| `self.ser.close()` | `self.transport.disconnect()` |
| `serial.Serial(port, ...)` | `SerialTransport(port, ...)` |

### 项目逻辑分离

所有项目级操作（哈希增量同步、状态比对、批量拉取）已从 `Flash.py` 迁移到 `cli/project/sync.py`：

| 迁移方法（原在 MicroPython 上） | 现位于 `ProjectSyncManager` |
|----------------------------------|-----------------------------|
| `project_scan()` | `scan()` |
| `project_flash()` | `flash()` |
| `project_status()` | `status()` |
| `project_pull()` | `pull()` |
| `_collect_project_files()` | 静态方法 / `_collect_project_files()` |
| `_check_device_files()` | `_check_device_files()` |
| `_discover_device_files()` | `_discover_device_files()` |
| `_compute_file_hash()` | 模块级 `compute_file_hash()` |

`ProjectSyncManager` 包装 `MicroPython` 实例，通过其公开 API 工作：

```python
from .project.sync import ProjectSyncManager

mp = MicroPython(port="COM3")
mp.connect()
syncer = ProjectSyncManager(mp)
syncer.scan("./my_project")
syncer.flash("./my_project", "/app")
syncer.status("./my_project", "/app")
syncer.pull("./my_project", "/")
```

`Flash.py` 不再导入 `json`、`hashlib`、`HASH_CONFIG_FILE`、`_HASH_VERSION`。

## WebREPLMicroPython (`utils/webrepl_micropython.py`)

一个轻量 `MicroPython` 子类，使用 `WebREPLTransport` 替代 `SerialTransport`：

```python
class WebREPLMicroPython(MicroPython):
    def __init__(self, url, password=None, timeout=10, transport=None):
        t = transport or WebREPLTransport(url, password)
        super().__init__(port=url, timeout=timeout, transport=t)
```

- 继承所有高级操作（`flash_file`、`fs_ls`、`run`、`reset` 等）
- 覆盖 `connect()` 执行 WebSocket 握手（忽略串口特定的 port/baudrate）
- `Flash.py` 完全不感知 WebREPL —— 分离由传输层强制执行
