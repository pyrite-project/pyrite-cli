# 设备刷入与项目同步

深入介绍 pyrite-cli 与 MicroPython 设备的通信机制、文件刷入协议、批量刷入、增量同步和设备文件管理。适合需要理解刷入原理和使用 project 系列命令的用户。

> 前置阅读：[快速上手](快速上手.md) — 了解基本命令和配置

---

## 1. 通信基础

### 1.1 UART 原始 REPL 协议

pyrite-cli 通过串口的 **原始 REPL 模式** 与 MicroPython 设备通信。原始 REPL 是 MicroPython 提供的一种二进制协议模式，允许主机端发送 Python 代码并将 stdout 逐字返回。

控制字符：

| 字符 | 值 | 作用 |
|------|-----|------|
| `Ctrl+A` | `\x01` | 进入原始 REPL 模式 |
| `Ctrl+B` | `\x02` | 退出原始 REPL，回到普通 REPL |
| `Ctrl+C` | `\x03` | 中断正在执行的程序 |
| `Ctrl+D` | `\x04` | 执行缓冲区中的代码 |
| `Ctrl+E` | `\x05` | 进入粘贴模式 |

每次设备操作的典型生命周期：

```
connect()            → 打开串口
_enter_raw_repl()    → Ctrl+C 中断 → Ctrl+A 进入原始 REPL
_operate()           → 执行代码 / 传输文件
_exit_raw_repl()     → Ctrl+B 退出原始 REPL
disconnect()         → 关闭串口
```

### 1.2 kbd_intr 保护

二进制文件中可能包含值 `0x03`（即 Ctrl+C）的字节。MicroPython 默认将 `0x03` 视为 `KeyboardInterrupt` 信号，收到后会重启设备，导致刷入失败。

解决方法：刷入前在设备端执行 `micropython.kbd_intr(-1)` 禁用中断，刷入完成后恢复 `kbd_intr(3)`。

### 1.3 REPL 串口日志

调试刷入问题时，所有串口收发数据自动记录到 `./log/flash_YYYYMMDD_HHMMSS.log`。控制字符替换为可读名称（`\x01` → `<RAW>`、`\x03` → `<C>`、`\x04` → `<D>`），便于排查。

## 2. 单文件刷入

### 2.1 流程概览

```
连接 → 进入原始 REPL → 预处理（条件编译）→ 编译 .py → .mpy（可选）
    → 发送 FLASH 模板脚本（含文件大小）→ 分块发送原始字节
    → 设备端接收写入 → 校验 → 完成
```

### 2.2 FLASH 模板脚本

刷入单个文件时，PC 端向设备注入以下 Python 脚本。设备端代为执行，接收数据并写入文件：

```python
import sys,micropython
try:
    micropython.kbd_intr(-1)
    usb = sys.stdin.buffer
    sys.stdout.write('READY')
    f_size = FSIZE          # 替换为文件总字节数
    with open(FILE, 'wb'):   # FILE 替换为设备上目标路径
        while f_size:
            want = min(BFSIZE, f_size)   # BFSIZE 替换为 chunk_size
            d = b''
            while len(d) < want:
                c = usb.read(min(64, want - len(d)))
                if c:
                    d += c
            f.write(d)
            f.flush()
            f_size -= len(d)
            sys.stdout.write('+')
    micropython.kbd_intr(3)
    rec = b''
    while not (b'ok' in rec):
        rec = (rec + (usb.read(1) or b''))[-16:]
    sys.stdout.write('ok')
except Exception as e:
    sys.stdout.write('FLASH_ERR:' + str(e) + '\\n')
    raise
```

设备端协议：发送 `READY` 表示就绪 → 每块写入后发送 `+` 表示进度 → 接收完毕确认 `ok`。

> **内层循环说明**：设备端每次最多请求 64 字节（MicroPython REPL 缓冲区的安全阈值），循环累积直到凑足 `want` 后写入文件。这比直接请求大块更可靠，避免缓冲区溢出。

### 2.3 分块传输

PC 端按 `chunk_size`（默认 4096 字节，由 `.pyrite_config.json` 配置）将文件切分为数据块，逐块发送。设备端每块写入后打印剩余大小。

| chunk_size | 优劣势 |
|-----------|--------|
| 512–1024 | 内存占用低，适合 RAM 紧张的设备；REPL 往返多，速度慢 |
| 4096–8192 | 往返少、速度快；需要设备有足够缓冲区 |

### 2.4 校验机制

刷入后的校验模式由配置中的 `verify` 字段控制：

| 模式 | 行为 | 适用场景 |
|------|------|----------|
| `off` | 不校验 | 速度优先，刷入量小 |
| `size` | 比对文件大小 | 默认模式，快速检测明显失败 |
| `crc32` | 比对文件大小 + CRC32 | 严格校验，检测数据损坏 |

校验失败时自动重试（次数由 `max_retries` 配置）。

### 2.5 刷入前预检查

`flash`、`flash-program` 和 `project flash` 会在真正写入设备前执行代码预检查，尽早拦截明显不会在 MicroPython 上运行的文件。

```bash
pyrcli flash COM3 main.py /main.py --check
pyrcli flash COM3 main.py /main.py --check=strict
pyrcli flash COM3 main.py /main.py --no-check
pyrcli project flash COM3 . /app --check
pyrcli flash-program COM3 src/ /app --check=strict --mp-version 1.20.0
```

默认 `basic` 模式会检查：

- `.py` 语法解析错误
- 编码声明或文件读取错误
- 条件编译 / manifest 过滤后的输出是否仍可解析
- 设备远程路径为空、包含 `..`、使用反斜杠或 Windows 盘符
- 同一批次内远程路径冲突

`strict` 模式会结合目标 MicroPython 版本检查常见语法兼容性问题。`precheck_compat` 控制兼容性问题是 `warn`、`error` 还是 `off`；`--mp-version` 可显式指定固件版本，未指定时会尽量从设备探测。

预检查只报告问题，不会自动改写用户源码；即使使用 `--dry-run`，预检查也会执行，但不会写入设备。特殊项目可用 `--no-check` 完全跳过。

## 3. 批量刷入

### 3.1 与单文件刷入的区别

`flash-program` 将整个目录树一次传输到设备，无需逐文件往返：

| | 单文件刷入 | 批量刷入 |
|--|-----------|----------|
| 脚本注入次数 | 每个文件一次 | 一次脚本注入 |
| 数据流 | 分文件独立传输 | 连续字节流 |
| 编译方式 | 单文件编译 | 多线程并行编译 |

### 3.2 三阶段流程

**阶段一：收集与预处理**

收集 `.py` 文件清单，应用条件编译预处理（如果指定了 active_tags），过滤 manifest.py 和 `.pyi` 文件。

**阶段二：并行编译**

使用 `ThreadPoolExecutor` 多线程并发调用 mpy-cross，将 `.py` 编译为 `.mpy`。编译失败的文件静默回退到原始 `.py`，不影响整体流程。

**阶段三：批量写入**

设备端创建目录结构 → 发送 `FLASH_PROGRAM` 脚本 → 连续流式传输所有文件字节。

### 3.3 FLASH_PROGRAM 模板脚本

```python
import sys, micropython

# 禁用 Ctrl+C 中断，防止数据流中的 0x03 字节误触发设备重启
try:
    micropython.kbd_intr(-1)
    usb = sys.stdin.buffer
    sys.stdout.write('READY')

    # FILES 格式: [(size, remote_path), ...]，运行前由 PC 端替换
    entries = FILES
    for file_size, file_path in entries:
        with open(file_path, 'wb') as f:
            remaining = file_size
            while remaining:
                blk = min(remaining, BFSIZE)
                d = usb.read(blk)
                if d:
                    f.write(d)
                    f.flush()
                    remaining -= len(d)
                    sys.stdout.write('+')
    # 恢复 Ctrl+C 中断
    micropython.kbd_intr(3)
except Exception as e:
    sys.stdout.write('FLASH_ERR:' + str(e) + '\\n')
    raise
```

PC 端将所有文件内容拼接连贯字节流，设备端按 `FILES` 中的尺寸依次切分写入，每块写入后发送 `+` 表示进度。

### 3.4 Manifest 清单系统

`manifest.py` 控制批量刷入时哪些文件进入设备，支持按 features 过滤。

语法：

```python
module("main.py")                                # 单个文件，无条件
module("lib/utils.py", features=["wifi"])        # 仅 wifi tag 激活时刷入
package("lib/drivers")                           # 递归目录下所有 .py
package("lib/net", features=["wifi"])            # 仅 wifi 时刷入整个目录
```

- `module(filename, remote=None, features=None)` — 单个文件
- `package(dirname, remote=None, features=None)` — 递归目录

参数：
- `remote`：设备上的目标路径，默认同 `filename`
- `features`：tag 列表，任意一个在 active_tags 中即包含；`None` 表示无条件包含

Manifest 使用安全 AST 解析器（基于 `ast` 模块，非 `exec`），只允许 `module()` 和 `package()` 调用，参数必须为字面量。

> Manifest 与源码条件编译可以联合使用，详见[条件编译实战 —— manifest 联合过滤](条件编译实战.md#6-manifest-联合过滤)

## 4. 增量同步（project 命令）

### 4.1 设计动机

在大项目或频繁迭代的场景下，每次都全量刷入所有文件效率低。`project` 系列命令通过 SHA256 哈希比对，**仅刷入新增或变更的文件**。

### 4.2 哈希配置

哈希配置保存在项目根目录的 `pyrite_file_config.json` 中：

```json
{
  "version": 1,
  "hash_algorithm": "sha256",
  "files": {
    "main.py": "e3b0c44298fc...",
    "lib/utils.py": "01ba4719c80b..."
  }
}
```

### 4.3 project hash — 离线扫描

扫描项目目录，计算所有可刷入文件的 SHA256 哈希并保存到配置文件。无需串口连接。

```bash
pyrcli project hash .                  # 默认保存在当前目录
pyrcli project hash ./src --manifest manifest.py  # 按 manifest 收集
```

### 4.4 project flash — 增量刷入

加载哈希配置 → 比对当前哈希 → 仅刷入新增或已变更的文件 → 更新哈希配置。

```bash
pyrcli project flash COM3 . /                    # 增量刷入
pyrcli project flash COM3 . / --target ESP32     # 指定 target
pyrcli project flash COM3 . / --feature sdcard   # 追加 feature
```

### 4.5 project status — 差异预览

比对本地哈希和设备端文件大小，只显示差异，不实际刷入：

```
[ADD] lib/utils.py        # 本地有，设备无
[MOD] main.py             # 本地哈希变化
[DEL] old_config.py       # 项目已删除
```

```bash
pyrcli project status COM3 . /
```

### 4.6 project pull — 从设备拉取

将设备上的文件批量下载到本地：

```bash
pyrcli project pull COM3 . /           # 下载全部
pyrcli project pull COM3 . / --dry-run # 预览模式
```

如果本地目录为空，自动从设备递归发现文件清单。传输使用单次脚本注入 + 批量传输，与 `flash-program` 流程对称。

### 4.7 project dev — 保存即刷入的开发会话

`project dev` 会持续监听本地项目，文件稳定后只刷入变化文件，并按配置进入或保持交互 REPL。

```bash
pyrcli project dev COM3 . /app
pyrcli project dev COM3 . /app --run --map-traceback
pyrcli project dev COM3 . /app --deep
pyrcli project dev COM3 . /app --once --no-repl
```

常用选项：

| 选项 | 说明 |
|------|------|
| `--run` / `--no-run` | 每次成功刷入后是否软重启运行 `boot.py` / `main.py` |
| `--no-repl` | 只监听和刷入，不进入交互 REPL |
| `--map-traceback` | 将设备 traceback 路径映射回本地文件 |
| `--lens` | 在 traceback 后展开本地源码上下文并标记出错行 |
| `--open-editor` | 与 `--lens` 配合，尝试用 VS Code 或 `$EDITOR` 打开首个匹配位置 |
| `--once` | 执行一轮同步后退出，适合本地 smoke 或 CI |
| `--poll-interval` / `--debounce` | 控制轮询频率和文件变化稳定等待时间 |
| `--on-error continue|stop|keep-repl` | 控制刷入或测试失败后的会话行为 |

`--deep` 是开发会话的组合模式，默认启用刷入后运行和 traceback 映射。需要更精确控制时，继续使用上面的细粒度选项。

traceback 映射会优先使用 `manifest.py` 的 `remote=` 信息；没有 manifest 时使用保守后缀匹配。`.py` 文件可显示本地行号，`.mpy` traceback 只提示源文件位置和 bytecode 降级信息，不伪造本地源码行号。

### 4.8 设备端测试与 test-on-save

`pyrcli test` 会把本地测试文件上传到设备临时目录执行，并解析 `PYRITE_TEST` 前缀结果帧。

```bash
pyrcli test COM3 test_device/ --timeout 15
pyrcli test COM3 test_device/ --keep-files
pyrcli project dev COM3 . /app --test-on-save=all --test-path test_device/
pyrcli project dev COM3 . /app --test-on-save=changed --lens
pyrcli project dev COM3 . /app --once --no-repl --test-on-save=all
```

测试中的普通 `print()` 会作为设备普通输出保留；host 侧只消费测试结果帧。`project dev --test-on-save` 会在保存并成功增量刷入后运行设备端测试，测试期间暂停新的刷入队列，避免 REPL、测试和刷入同时占用 Raw REPL。`--once --test-on-save=all` 可作为无交互 smoke，测试失败时退出码为 1。

### 4.9 snapshot — 设备文件系统快照与回滚

快照适合在大规模刷入、尝试 WebDAV 编辑、调试第三方库前保存当前设备状态。

```bash
pyrcli snapshot save COM3 before-change --remote-path /app
pyrcli snapshot list
pyrcli snapshot diff COM3 before-change --remote-path /app
pyrcli snapshot restore COM3 before-change
pyrcli snapshot restore COM3 before-change --apply --yes
pyrcli project flash COM3 . /app --snapshot-before before-flash
```

快照保存在本地 `.pyrite_snapshots/<name>/`，包含文件内容、设备路径、大小和 SHA256。`restore` 默认只输出 dry-run 计划；真正恢复必须显式加 `--apply`，无人值守场景再加 `--yes`。未传 `--remote-path` 时，恢复命令默认只扫描快照中文件的共同父目录，避免把无关设备路径纳入删除计划。

`diff` / `restore` 对比阶段在 host 侧下载文件并计算 SHA256，避免依赖旧固件上不完整或缺失的 `hashlib.sha256`。可用 `--include`、`--exclude` 和 `--max-file-bytes` 收窄快照范围。

## 5. 设备文件管理

### 5.1 Bytes 下载协议

从设备读取文件时采用两阶段协议：

1. 通过 `run()` 获取文件大小（文本行）
2. 在 Raw REPL 中执行字节输出脚本，PC 端按已知大小直接接收

```python
# 设备端执行的大致脚本
import sys
size = os.stat(path)[6]
sys.stdout.write(str(size) + "\n")
sys.stdout.buffer.write(open(path, 'rb').read())
```

### 5.2 fs ls 增强

```bash
pyrcli fs ls COM3 /              # 基本列表
pyrcli fs ls COM3 / -r           # 递归列出全部文件
pyrcli fs ls COM3 / --sort size  # 按大小排序（默认按文件名）
pyrcli fs ls COM3 / -p           # 分页，每页 20 行
```

`fs_ls_recursive()` 在设备端通过递归 `os.listdir()` + `os.stat()` 一次性遍历目录树，避免多次往返。

### 5.3 MicroPython 包安装

`pyrcli pkg` 是对 `mpremote mip install` 的薄封装，适合在同一个 CLI 中完成包安装计划、离线包校验和安装执行。

```bash
pyrcli pkg install COM3 aioble --target /lib --dry-run
pyrcli pkg install COM3 github:org/repo@branch --target /lib
pyrcli pkg cache aioble --version latest --dry-run
pyrcli pkg install-offline COM3 .pyrite/pkg-cache/aioble
```

`pkg install` 构建并执行 `mpremote connect <port> mip install <package>`；`pkg install-offline` 校验本地 `package.json` 或包含它的目录后再委托 `mpremote` 安装；`pkg cache` 当前只规划缓存路径并审计本地包描述，不自行联网下载完整依赖。

### 5.4 GPIO 监控

`monitor` 是只读采样工具，用于周期性查看 GPIO 输入状态。

```bash
pyrcli monitor COM3
pyrcli monitor COM3 --pins 0,2,4,5
pyrcli monitor COM3 --interval 0.2 --format json --count 20
pyrcli monitor COM3 --pins 0,2 --edge changed --duration 10
```

未指定 `--pins` 时，设备端会探测保守 GPIO 候选列表；探测失败时提示显式传入引脚。采样只使用 `machine.Pin(pin, machine.Pin.IN)`，不会设置上下拉或输出模式。不同板卡的保留引脚、启动绑带引脚差异较大，生产排查建议显式指定 `--pins`。

### 5.5 主机能力透传

`tunnel` 让设备端脚本通过 REPL 借用上位机能力，定位是开发、调试、演示和临时验证，不替代正式生产协议。

```bash
pyrcli tunnel kb COM3
pyrcli tunnel network COM3 --allow example.com
pyrcli tunnel network COM3 --allow 127.0.0.1 --allow-private
pyrcli tunnel network COM3 --ws ws://192.168.4.1:8266/ --allow example.com --allow-webrepl
```

`tunnel kb` 透传普通字符、回车、退格和方向键等键盘事件，保留 `Ctrl+C` / `Ctrl+]` 作为主机侧退出组合。`tunnel network` 以上位机作为受限 HTTP(S) 跳板，必须显式传入 `--allow` 白名单；访问 localhost、私有地址或链路本地地址还需要 `--allow-private`。WebREPL 连接下启动 network tunnel 需要额外确认 `--allow-webrepl`，避免网络通道叠加时权限边界不清晰。

## 6. 调试与错误处理

| 场景 | 表现 |
|------|------|
| 串口不存在/被占用 | `serial.SerialException` |
| 设备在原始 REPL 中返回 Traceback | `RuntimeError`，输出完整 Traceback |
| 刷入中途断开 | `serial.SerialException`，设备端文件可能未关闭 |
| 本地文件不存在 | `FileNotFoundError` |
| 目录无效 | `NotADirectoryError` |
| 无法进入原始 REPL | `RuntimeError`，含设备原始响应 |
| 设备文件读取超时/不完整 | `RuntimeError`，含已接收字节数 |
| 无法识别设备 target | 提示使用 `--target` 手动指定 |

### 6.1 Flight Recorder trace

`flash --trace` 会记录一次刷入过程的阶段、串口/WebSocket 流量、控制字符和失败信息，适合排查断线、校验失败或 Raw REPL 初始化失败。

```bash
pyrcli flash COM3 main.py /main.py --trace
pyrcli flash COM3 main.py /main.py --trace --trace-path .pyrite-trace/main.trace.jsonl
```

trace 输出会把控制字符转为 `<RAW>`、`<C>`、`<D>` 等可读文本，并脱敏 password、token、Authorization 等敏感字段。提交问题时优先附带 trace 文件，比只贴最终异常更容易复现。

### 6.2 debug doctor

`debug doctor` 用轻量探针查看设备固件能力，例如 `sys.settrace`、`micropython.kbd_intr`、`network`、`webrepl`、文件系统读写和 Python 语法支持。

```bash
pyrcli debug doctor COM3
```

doctor 的输出可辅助选择降级路径：设备缺少 `network` 时可尝试 `tunnel network` 做开发期验证；固件版本可作为 strict 预检查的兼容性依据；某些调试能力缺失时，`project dev` 仍会保留基础刷入和 REPL 路径。

### 6.3 串口占用处理

当串口打开失败且错误看起来像“权限不足 / 已占用 / resource busy”时，交互式终端会询问是否扫描占用进程。Windows 优先使用 Sysinternals `handle.exe` 查找句柄，找不到时退回启发式命令；Linux/macOS 使用 `lsof`，不可用时尝试 `fuser`。

结束进程前会显示 PID、进程名和来源，并再次确认；非交互式环境会跳过该流程，避免 CI 或脚本中意外结束其他进程。
