# pyrite-cli v0.0.5 更新预览

v0.0.5 的目标是把 pyrite-cli 从“刷入工具”继续推进到“开发会话工具”：围绕 MicroPython 设备的代码同步、运行、调试、文件浏览和外部能力接入，减少在串口工具、文件管理器、浏览器、编辑器之间来回切换的成本。

本次更新仍以 Raw REPL / UART 稳定链路为主，继续兼容 WebREPL。新增能力优先做成可选开关，不改变 `flash`、`flash-program`、`project flash`、`mount` 等已有稳定路径的默认行为。

## 更新主线

| 模块 | 目标 | 用户价值 |
| --- | --- | --- |
| `tunnel` | 让设备通过 REPL 借用上位机输入与网络能力 | 无 WiFi、旧固件、临时外设不足时也能快速验证想法 |
| `project dev --deep`（已完成） | 打通监听、增量刷入、运行、traceback 映射 | 让开发循环更接近“保存即刷入，报错能定位本地源码” |
| 代码预检查（已完成） | 刷入前拦截明显的 Python 语法和兼容性问题 | 避免把必然失败的代码传到设备上浪费时间 |
| `mount` 缓存增强（已完成） | 提升 WebDAV 文件浏览和编辑体验 | 文件管理器访问更顺滑，保存后的目录状态更一致 |
| 调试可观测性（已完成） | 增强 trace、doctor、device test 的版本联动 | 问题可复现、可归档、可自动测试 |
| `project dev --lens`（已完成） | traceback 直接展开本地代码片段 | 报错时不用再手动找文件和行号 |
| `project dev --test-on-save`（已完成） | 保存刷入后自动跑设备端测试 | 把“改代码 -> 刷入 -> 验证”变成一条链 |
| `snapshot`（已完成核心能力） | 设备文件系统快照与一键回滚 | 大胆试错，失败后快速回到已知状态 |

## 新增亮点功能

以下 3 条偏向“看一眼就能感受到开发体验提升”的能力。每一项都保持可选，不影响现有刷入和 REPL 默认路径。

### 1. `project dev --lens`：本地代码透镜

在 `--map-traceback` 的基础上进一步增强：当设备输出 traceback 时，CLI 不只显示路径映射，还直接展开本地源码附近的代码片段，并高亮出错行。

目标效果：

```text
Traceback (most recent call last):
  File "/app/lib/sensor.py", line 24, in read
/app/lib/sensor.py:24 -> lib/sensor.py:24

lib/sensor.py
  21 |     def read(self):
  22 |         raw = self.bus.readfrom_mem(self.addr, 0x00, 2)
  23 |         value = raw[0] << 8 | raw[1]
> 24 |         return scale(value, self.range)
  25 |
NameError: name 'scale' isn't defined
```

已实现能力：

- `--lens` 覆盖并增强 `--map-traceback` 体验；原 `--map-traceback` 仍可单独使用。
- 对 `.py` traceback 展示本地代码上下文，默认前后各 3 行。
- 对 `.mpy` traceback 只显示源文件位置和 bytecode 降级提示，不伪造本地源码行号。
- 可选 `--open-editor`，在支持的环境中用 VS Code 或 `$EDITOR` 打开首个匹配文件位置，失败不终止 dev 会话。

命令：

```bash
pyrcli project dev COM3 . /app --lens
pyrcli project dev COM3 . /app --lens --open-editor
```

验收口径：

- 纯逻辑测试覆盖代码片段截取、首尾行边界、`.mpy` 降级提示。
- 未匹配 traceback 时保持原始输出，不注入误导信息。
- Windows 路径、MSYS2 路径和 manifest `remote=` 映射都要覆盖。

### 2. `project dev --test-on-save`：保存后自动设备测试

在开发会话中，文件保存并成功增量刷入后，自动运行设备端测试或指定 smoke 脚本，把验证动作并入同一个循环。

目标体验：

```text
[dev] 刷入结束，REPL 可用
[test] running 3 device tests...
[test] PASS drivers/test_i2c.py  18ms
[test] FAIL app/test_sensor.py   AssertionError: expected 42
```

已实现能力：

- 默认发现 `test_device/`，复用现有 `pyrcli test` 的上传、执行、解析和清理逻辑。
- 支持 `--test-path` 指定测试目录或单个测试文件。
- 支持 `--test-on-save=all|changed|off`，其中 `changed` 只跑与变更文件相关的测试候选。
- 测试失败时继续留在 REPL；失败 traceback 走同一套 `--lens` / `--map-traceback` 输出。

命令：

```bash
pyrcli project dev COM3 . /app --test-on-save=all
pyrcli project dev COM3 . /app --test-on-save=changed --test-path test_device/
```

验收口径：

- 纯逻辑测试覆盖测试发现、changed 匹配、失败结果解析。
- 设备连接独占，测试运行期间暂停文件刷入队列，避免 REPL/测试/刷入同时抢 Raw REPL。
- `--once --test-on-save=all` 可用于本地 CI smoke；测试失败时退出码为 1。

### 3. `snapshot`：设备文件系统快照与一键回滚

提供一个面向试错的设备文件系统快照功能。刷入大型改动、尝试 `mount` 编辑、调试第三方库前，先保存当前设备状态；失败后可以一条命令恢复。

命令：

```bash
pyrcli snapshot save COM3 before-display-refactor
pyrcli snapshot list
pyrcli snapshot diff COM3 before-display-refactor
pyrcli snapshot restore COM3 before-display-refactor
pyrcli snapshot restore COM3 before-display-refactor --apply --yes
```

已实现核心能力：

- 快照保存到本地 `.pyrite_snapshots/<name>/`，包含文件内容、设备路径、大小和 SHA256。
- `restore` 默认先 dry-run 显示将新增、覆盖、删除的文件；执行恢复需要显式 `--apply`，跳过交互确认需要再加 `--yes`。
- 支持 `--include` / `--exclude` 过滤目录，默认跳过缓存、日志和超大文件。
- `diff` / `restore` 对比阶段在 host 侧下载文件并计算 SHA256，避免依赖旧固件不完整或缺失的 `hashlib.sha256`。
- `restore` 支持 `--remote-path` 收窄扫描范围；未显式指定时默认使用快照中文件的共同父目录，避免把无关设备路径纳入删除计划。
- 与 `project flash` 联动：可选 `--snapshot-before` 在刷入前自动创建快照。

验收口径：

- 纯逻辑测试覆盖快照 manifest、diff 计划、restore 计划。
- 实机验证至少覆盖保存、列出、diff、恢复小型目录。
- 恢复命令必须明确提示会覆盖设备文件，避免误操作。

## 1. `tunnel` 命令：主机能力透传

### 背景

很多 MicroPython 板卡的限制不是代码本身，而是临时缺少输入、网络或证书/协议支持。例如：

- 部分开发板没有 WiFi，或当前固件没有启用 `network`。
- 旧固件的 `urequests`、TLS、DNS 行为不稳定。
- 设备端不方便接键盘、旋钮、传感器替身等交互输入。
- 开发阶段只想快速验证业务逻辑，不想先完整移植所有 I/O 能力。

`tunnel` 计划提供一条“设备请求，上位机代办”的轻量通道。它不替代正式生产协议，只服务开发、调试、演示和临时验证。

### 命令草案

```bash
pyrcli tunnel kb COM3
pyrcli tunnel network COM3 --host 127.0.0.1 --allow example.com
pyrcli tunnel network COM3 --ws ws://192.168.4.1:8266/ --password ***
```

### `tunnel kb`

`tunnel kb` 将上位机键盘输入映射给设备端脚本，设备端通过一个小型 helper 读取输入事件。

已实现能力：

- 支持普通字符、回车、退格、方向键等常用键位。
- 支持阻塞读取和非阻塞轮询两种模式，便于 REPL demo 和主循环程序使用。
- Windows 优先使用 `msvcrt`，类 Unix 系统使用终端 raw mode。
- 保留 `Ctrl+C` / `Ctrl+]` 一类退出组合，不让设备端脚本吞掉主机侧中断。

设备端 helper 方向：

```python
from pyrite_tunnel import read_key, poll_key

while True:
    key = poll_key()
    if key:
        print("key:", key)
```

### `tunnel network`

`tunnel network` 以上位机作为网络跳板，由设备端发出简化请求，上位机执行 HTTP(S) 请求后把状态码、响应头摘要和响应体返回给设备。

计划能力：

- 覆盖 `GET`、`POST`、`PUT`、`DELETE` 的基本场景。
- 支持超时、最大响应大小、允许域名白名单、禁止内网探测等安全限制；未配置 `--allow` 时拒绝请求。
- 默认只面向开发用途，不保存 cookie，不自动转发主机环境中的敏感 header。
- 对大响应优先支持分块读取，避免设备端一次性申请过大内存。

WebREPL 说明：

- `tunnel` 总体支持 `--ws` 连接方式。
- `tunnel network` 在 WebREPL 模式下默认关闭或要求显式确认，避免“网络通道套网络通道”导致权限边界不清晰。

### 脚本位置

设备端脚本计划放在：

```text
cli/device/tunnel_scripts/
```

其中主机侧 CLI 只负责注入、握手、转发和日志；设备端 helper 保持短小，避免增加固件侧依赖。

旧固件兼容性：`network.py` 和 `kb.py` 的设备端 helper 不再直接假定 `sys.stdout.flush()` 存在；在 MicroPython 1.12.0 等 `sys.stdout` 为 `FileIO` 的固件上，也能完成 hello 帧输出并继续会话。

### 验收口径

- 无设备纯逻辑测试：请求/响应帧编码、键盘事件编码、超时和白名单校验。
- CLI 冒烟：`pyrcli tunnel --help`、`pyrcli tunnel kb --help`、`pyrcli tunnel network --help`。
- 实机验证：ESP32/ESP8266 至少完成一次键盘输入 demo 和一次小型 HTTP GET demo。

## 2. `project dev --deep`：深度开发会话（已完成）

### 背景

`project dev` 已经承担“监听项目变化、增量刷入、进入 REPL”的职责。`--deep` 方向是把它扩展为更完整的开发会话：

1. 启动时同步项目。
2. 文件保存后只刷入变化文件。
3. 可选择刷入后软重启运行 `boot.py` / `main.py`。
4. 设备 traceback 输出自动映射回本地源码。
5. REPL 仍可交互，不把开发者锁死在自动流程里。

现有实现中已经存在的拆分能力包括：

- `--run`：每次成功刷入后软重启。
- `--no-repl`：只监听和刷入，不进入交互 REPL。
- `--map-traceback`：将设备路径映射回本地项目路径。
- `--once`：执行一轮同步后退出，便于测试和 CI。
- `--poll-interval` / `--debounce`：控制文件监听频率和稳定等待。
- `--on-error continue|stop|keep-repl`：控制错误处理策略。

`--deep` 可以作为组合开关，也可以继续保持这些细粒度选项。建议先保留细粒度选项，等实际使用稳定后再决定是否加入 `--deep` 作为快捷别名。

### traceback 映射

目标效果：

```text
  File "/app/lib/sensor.py", line 24, in read
/app/lib/sensor.py:24 -> lib/sensor.py:24
```

对于自动编译后的 `.mpy`：

```text
  File "/app/lib/sensor.mpy", line 3, in read
/app/lib/sensor.mpy:3 -> lib/sensor.py (.mpy bytecode; source line unavailable)
```

设计边界：

- `.py` 文件可以保留本地行号。
- `.mpy` traceback 不伪造本地行号，只提示源文件位置。
- 优先使用 `manifest.py` 的 `remote=` 重映射；没有 manifest 时使用保守后缀匹配。
- 多个本地文件都可能匹配同一设备路径时不强行猜测，避免误导。

### 运行与 REPL 行为

计划保持以下交互规则：

- 文件变化期间提示 `REPL 暂不可用`，刷入完成后提示 `REPL 可用`。
- REPL 中输入 `:run` 或 `run` 可触发软重启。
- `--run` 成功刷入后自动软重启，失败时按 `--on-error` 策略处理。
- `manifest.py`、`.pyrite_config.json`、`pyproject.toml` 变化会触发全量重新计算；普通 `.py` 变化优先走快速增量路径。

### 验收口径

- `test/test_project_dev.py` 覆盖 watcher、增量 changed paths、`:run`、busy/ready 提示和 traceback mapper 注入。
- `test/test_traceback_map.py` 覆盖 manifest 远程映射、`.mpy` 映射、未匹配输出保持不变。
- CLI 冒烟：`pyrcli project dev --help`。

## 3. 功能增强：代码预检查（已完成）

### 背景

当前刷入流程会把代码传到设备后再由设备解释执行。对于明显语法错误、条件编译后残留的无效代码、或目标固件不支持的语法，这会浪费串口传输时间，也会把错误暴露在更难定位的设备端。

v0.0.5 计划在刷入前加入可配置的预检查阶段。

### 检查范围

默认检查：

- `.py` 文件语法解析失败。
- 编码声明或文件读取错误。
- 条件编译 / manifest 过滤后的输出是否仍能解析。
- 明显的空文件、路径不合法、远程路径冲突。

可选严格检查：

- 根据目标固件版本和 board profile 拦截不兼容语法。
- 对 f-string、类型注解、`match/case`、walrus 等语法做版本规则判断。
- 对被条件编译禁用的函数调用给出 warning 或 error。

不建议默认做的事：

- 不默认自动改写用户源码。
- 不把 CPython 静态分析结果当成 MicroPython 运行时结论。
- 不执行用户代码来“检查导入”，避免副作用。

### 命令与配置草案

```bash
pyrcli flash COM3 main.py /main.py --check
pyrcli project flash COM3 . /app --check
pyrcli project flash COM3 . /app --check=strict
pyrcli project flash COM3 . /app --no-check
```

配置草案：

```json
{
  "precheck": "basic",
  "precheck_compat": "warn"
}
```

### f-string 兼容策略

原草案提到“必要时替换为 `.format()`”。这里建议拆成两步：

1. 默认只报告问题和建议写法，不改用户文件。
2. 若以后要自动改写，必须通过显式命令或 `--fix` 开关执行，并保留 diff 预览。

这样可以避免工具在刷入路径中静默改动源码。

### 验收口径

- 纯逻辑测试覆盖语法错误、条件编译后错误、严格模式 warning/error 切换。
- 错误信息必须显示本地文件路径和行列号。
- `--dry-run` 下也执行预检查，但不连接设备。
- `--no-check` 能完全跳过该阶段，便于兼容特殊项目。

## 4. 性能优化：`mount` 缓存增强（已完成）

### 背景

`mount` 通过 PC 侧 WebDAV 服务把设备文件系统暴露给系统文件管理器。文件管理器会频繁发出 `PROPFIND`、`GET`、`PUT`、`MOVE`、`DELETE` 等请求，如果每次都直连设备查询，会明显受串口延迟影响。

v0.0.5 的 `mount` 缓存增强重点是“目录和 stat 信息缓存”，而不是默认缓存完整文件内容。

### 计划能力

- 首次访问目录时缓存目录项和文件 stat。
- `PUT` 保存文件后同步更新对应 stat 和父目录缓存。
- `MKCOL`、`DELETE`、`MOVE`、`COPY` 后精确更新或失效相关缓存。
- `--load-all` 可在启动时递归预热完整目录树。
- 启动期根目录短暂为空时返回重试，避免文件管理器误以为设备是空盘。
- `mount-run` 执行设备脚本期间暂停文件通道，写请求按队列上限暂存，脚本结束后再回放。

### 缓存边界

- 默认不缓存完整文件内容，避免大文件占用主机内存，也避免设备端被外部 REPL 修改后读取陈旧内容。
- 缓存只承诺提高文件管理器交互体验，不作为设备文件系统的权威状态。
- 用户通过其他工具直接修改设备文件后，可能需要刷新目录或重启 `mount` 会话。

### 验收口径

- `test/test_webdav_mount.py` 覆盖目录缓存、空目录重试、run 队列、删除/移动/复制后的状态同步。
- CLI 冒烟：`pyrcli mount --help`、`pyrcli mount-run --help`。
- 实机验证：Windows 文件管理器至少完成浏览、打开、保存、重命名、删除、运行 `_run.bat`。

## 5. 调试可观测性联动（已完成）

v0.0.5 可以把已有调试能力串成一套定位流程，而不是散落在不同命令里。

### Flight Recorder trace

`flash --trace` 记录一次刷入过程的阶段、流量、控制字符和失败信息，`trace view` / `trace summarize` 用于查看和归档。

计划补强：

- 在 v0.0.5 文档中推荐断线、校验失败、Raw REPL 失败时优先附带 `.pyrite-trace`。
- trace 输出继续脱敏 password、token、Authorization 等敏感字段。
- 保持控制字符可读化，例如 `<RAW>`、`<C>`、`<D>`。

### `debug doctor`

`debug doctor` 可以作为设备能力探测入口，输出 `sys.settrace`、`micropython.kbd_intr`、`network`、`webrepl` 等 feature 状态。

计划补强：

- 与 `tunnel network` 关联：设备无 `network` 时提示可尝试 tunnel。
- 与预检查关联：设备固件版本可用于兼容规则。
- 与 `project dev` 关联：缺少必要能力时给出降级说明。

### 设备端测试

`pyrcli test` 的定位是把本地 `test_device/` 上传到设备执行并回收结果。

已实现补强：

- 在 v0.0.5 验收中增加小型设备端 smoke test。
- 支持 `--keep-files` 保留测试文件，便于复现。
- 与 `project dev --once` 配合，用于 CI 或本地快速验证。
- 为兼容旧固件，测试 runner 不再替换 `sys.stdout`；测试中的普通 `print()` 保持为设备普通输出，host 侧只消费 `PYRITE_TEST` 前缀结果帧。

## 串口占用处理（已完成）

串口被 IDE、串口监视器或上一次异常退出的进程占用时，`pyrcli` 会在交互式终端中提供可选解除流程：先识别“权限不足 / 已占用 / resource busy”等打开失败，再询问是否扫描占用进程。

当前实现边界：

- Windows 优先使用 Sysinternals `handle.exe` 精确扫描句柄；未安装时退回命令行启发式扫描。
- Linux/macOS 使用 `lsof`，不可用时尝试 `fuser`。
- 结束进程前会展示 PID、进程名和来源，并再次要求用户确认；仍未退出时才询问是否强制结束。
- 非交互式环境会跳过该流程，避免 CI 或脚本中意外结束其他进程。

## 兼容性与稳定性原则

本次更新不应为了表面性能牺牲已知稳定性：

- 不默认放大设备端 `usb.read(min(64, ...))` 的读取窗口。
- 不默认移除批量刷入中的 host 侧 `all_data = b"".join(...)` 合并。
- 不默认缩短串口连接、DTR、RTS、Raw REPL 初始化等待。
- WebREPL 支持保持为可选路径，不能影响串口默认路径。
- 所有新增网络能力默认本机回环、最小权限、显式开启。
