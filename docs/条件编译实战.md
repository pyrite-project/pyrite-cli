# 条件编译实战

pyrite-cli 的条件编译系统允许你**一份源码管理多平台、多配置**的 MicroPython 固件构建。通过 `feature`/`target` 宏标记平台相关和可选功能代码，刷入前自动预处理，不匹配的代码被编译期消除。

> 前置阅读：[快速上手](快速上手.md) — 了解基础命令和配置
> 相关阅读：[设备刷入与项目同步 —— Manifest 清单系统](设备刷入与项目同步.md#34-manifest-清单系统)

---

## 1. 概念与动机

### 1.1 为什么需要条件编译

MicroPython 设备资源极其有限（Flash 通常 1–16 MB，RAM 低至 128 KB），无法像 CPython 那样运行时动态加载。条件编译让你在**刷入前**就决定哪些代码进入设备，避免运行时条件判断的开销和死代码占据空间。

处理流水线：

```
源码（含宏标记）        预处理（libcst）        编译（mpy-cross）
    │                   │                       │
    ▼                   ▼                       ▼
with target("ESP32"):   展开为 if True/False    死代码消除
@feature("ble")         行号完全对齐             仅保留匹配代码
```

### 1.2 target vs feature

| | target | feature |
|--|--------|---------|
| 语义 | 硬件平台（互斥） | 功能特性（可组合） |
| 来源 | 设备自动识别 / `--target` 指定 | board_tags 隐式激活 / `--feature` 追加 |
| 典型值 | `ESP32`、`RP2040`、`STM32` | `wifi`、`ble`、`sdcard`、`debug` |
| 互斥性 | 同一时刻只有一个 target | 多个 feature 可共存 |

两者在语法上完全一致，区别仅在语义约定。

### 1.3 active_tags 组成

```
active_tags = resolve_target(设备自动识别 | --target 手动指定)
            + target 隐式激活的 features（来自 board_tags 配置）
            + --feature 追加
            - --no-feature 移除
```

## 2. 语法参考

### 2.1 `with` 块

```python
with target("ESP32"):
    import network          # 仅 ESP32 编译此块

with feature("wifi"):
    connect_wifi()          # 仅 wifi tag 激活时编译
```

预处理后展开为 `if True:`（tag 匹配）或 `if False:`（tag 不匹配），行号完全对齐。

### 2.2 函数装饰器

```python
@target("ESP32")
def connect_wifi():
    ...

@target("RP2040")
def connect_wifi():
    ...
```

匹配的实现保留（装饰器移除），不匹配的实现包裹在 `if False:` 中：

```python
def connect_wifi():         # ESP32 匹配
    ...

if False:                   # RP2040 不匹配
    def connect_wifi():
        ...
```

### 2.3 装饰器叠加

多个装饰器叠加为逻辑 AND：

```python
@target("ESP32")
@feature("ble")
def ble_init():
    ...
```

仅 `target=ESP32` 且 `feature=ble` 同时满足时才保留。

### 2.4 IDE 类型存根

`pyrcli new` 创建项目时自动生成 `feature_stub.pyi`，让 IDE 不报 `feature`/`target` 未定义错误：

```python
class feature:
    def __init__(self, name: str) -> None: ...
    def __enter__(self) -> "feature": ...
    def __exit__(self, *a: object) -> None: ...
    def __call__(self, fn: object) -> object: ...

target = feature
```

`.pyi` 文件不会被 `flash-program` 收集，无需手动排除。

## 3. 静态警告分析

预处理时对每个文件进行两遍静态分析，向 stderr 输出黄色警告：

### 警告 1：函数所有实现均不匹配

```
[警告] main.py: 函数 'connect_wifi' 所有实现均不匹配当前 tags
```

触发条件：某函数名的全部 `@target/@feature` 实现均不在 `active_tags` 中。

### 警告 2：裸调用未被 guard 保护

```
[警告] main.py: 裸调用 'connect_wifi()' 未被 guard 保护，运行时将 NameError
```

触发条件：调用点不在任何 `with target/feature(...)` 块内，且被调函数已被全部禁用。

> 跨文件调用不检测，属于已知限制。

## 4. 实战场景

### 4.1 场景一：单源码支持多款开发板

ESP32-S3 和 RP2040 引脚定义、外设驱动不同。传统做法是维护两份代码，难以同步。

```python
# main.py
import time

# ── 引脚定义 ──
with target("ESP32"):
    from machine import Pin
    LED_PIN = 2
    I2C_SCL = 22
    I2C_SDA = 21

with target("RP2040"):
    from machine import Pin
    LED_PIN = 25
    I2C_SCL = 5
    I2C_SDA = 4

# ── 初始化（公共逻辑）──
led = Pin(LED_PIN, Pin.OUT)
```

刷入时工具自动识别 target：

```bash
pyrcli flash COM3 main.py /main.py      # 自动识别
pyrcli flash COM3 main.py /main.py --target ESP32   # 离线指定
```

使用装饰器实现多平台同名函数：

```python
@target("ESP32")
def init_sensors():
    import dht
    return dht.DHT22(Pin(15))

@target("RP2040")
def init_sensors():
    from machine import ADC
    return ADC(4)

sensor = init_sensors()    # 自动使用当前平台对应实现
```

### 4.2 场景二：按功能特性裁剪固件

不同客户需要不同的功能组合（WiFi、蓝牙、SD 卡、OLED），全部编译体积超标。

```python
# ── WiFi 功能 ──
with feature("wifi"):
    import network

    def upload_data(data):
        wlan = network.WLAN(network.STA_IF)
        wlan.active(True)
        wlan.connect(SSID, PASSWORD)

# ── SD 卡日志功能 ──
with feature("sdcard"):
    def log_to_sdcard(msg):
        with open("/sd/log.txt", "a") as f:
            f.write(msg + "\n")
```

刷入时灵活组合：

```bash
# 仅 WiFi 版本
pyrcli flash-program COM3 src/ --target ESP32

# WiFi + SD 卡版本
pyrcli flash-program COM3 src/ --target ESP32 --feature sdcard

# 全功能版本
pyrcli flash-program COM3 src/ --target ESP32 --feature sdcard,ble,oled

# 禁用某个特性
pyrcli flash-program COM3 src/ --target ESP32 --no-feature wifi
```

### 4.3 场景三：开发版 vs 生产版差异管理

开发时需要调试日志、模拟数据；生产版本去掉这些，加入看门狗、错误上报。

```python
# utils.py
with feature("debug"):
    def log(msg):
        print(f"[DEBUG] {msg}")

    def simulate_sensor():
        return {"temp": 25.0, "hum": 60.0}

with feature("prod"):
    import machine
    WDT_TIMEOUT = 30000

    def init_watchdog():
        wdt = machine.WDT(timeout=WDT_TIMEOUT)
        return wdt
```

一行命令切换：

```bash
# 开发模式
pyrcli flash-program COM3 src/ --feature debug

# 发布模式
pyrcli flash-program COM3 src/ --no-feature debug --feature prod
```

## 5. 常用设计模式

### 5.1 平台抽象层 (PAL)

将平台相关代码收拢到 `pal.py`：

```python
# pal.py — Platform Abstraction Layer
import machine

@target("ESP32")
def sleep_ms(ms):
    machine.sleep(ms)

@target("RP2040")
def sleep_ms(ms):
    import time
    time.sleep_ms(ms)

@target("ESP32")
def device_id():
    import esp32
    return ''.join(f"{b:02x}" for b in esp32.mac())

@target("RP2040")
def device_id():
    from machine import unique_id
    return ''.join(f"{b:02x}" for b in unique_id())
```

业务代码只调用 `pal.sleep_ms(100)`，零平台耦合。

### 5.2 功能模块开关（manifest 联合过滤）

将整块功能独立成文件，用 manifest.py 的 `features` 决定是否刷入：

```python
# manifest.py
module("main.py")
module("pal.py")
module("wifi_manager.py", features=["wifi"])
module("ble_service.py", features=["ble"])
package("drivers/", features=["sdcard"])
```

不匹配的文件根本不进入设备，实现文件级裁剪。

### 5.3 分层 feature 设计

用不同 feature 实现精细控制：

```python
with feature("log_minimal"):
    def log_error(msg): print(f"[E] {msg}")

with feature("log_verbose"):
    def log_error(msg):   print(f"[E] {msg}")
    def log_info(msg):    print(f"[I] {msg}")
    def log_debug(msg):   print(f"[D] {msg}")
```

```bash
# 按需选择日志级别
pyrcli flash COM3 main.py --feature log_verbose
pyrcli flash COM3 main.py --feature log_minimal  # 不加则无日志
```

## 6. manifest 联合过滤

manifest.py 的 `features` 与源码中的 `feature()` 块形成两层过滤：

| 层次 | 机制 | 效果 |
|------|------|------|
| 文件级 | manifest.py `features` 参数 | 不匹配的文件根本不刷入设备 |
| 代码级 | 源码中 `with feature()` / `@feature()` | 刷入的文件内，不匹配的代码块被消除 |

```python
# manifest.py
module("sensor_drivers.py", features=["sensor"])

# sensor_drivers.py
with feature("sensor_temperature"):
    class TemperatureSensor: ...

with feature("sensor_humidity"):
    class HumiditySensor: ...
```

刷入 `--feature sensor,sensor_temperature` 时：
- `sensor_drivers.py` 被刷入（匹配 `sensor`）
- 文件中仅 `TemperatureSensor` 保留，`HumiditySensor` 被消除

## 7. CLI 参数速查

| 参数 | 适用命令 | 说明 |
|------|----------|------|
| `--target TAG` | flash, flash-program, project flash | 手动指定 board target |
| `--feature TAG[,TAG]` | flash, flash-program, project flash | 追加激活的 tags |
| `--no-feature TAG[,TAG]` | flash, flash-program, project flash | 强制禁用的 tags |
| `--manifest PATH` | flash-program, project flash | 指定 manifest.py 路径 |

## 9. 常见陷阱

### 陷阱 1：调用点未被 guard

```python
@feature("wifi")
def connect_wifi(): ...

connect_wifi()          # ← 警告：裸调用，wifi 未激活时 NameError
```

**解决**：将调用点也放入 `with feature("wifi"):`。

### 陷阱 2：`with` 内嵌 return/break/continue

预处理后 `with target(...)` 变为 `if True/False:`，在 MicroPython 中 `if` 内的 `return`、`break`、`continue` 合法，但需要注意语义不变。

### 陷阱 3：跨文件引用未 guard 的函数

```python
# utils.py
@feature("wifi")
def scan(): ...

# main.py
from utils import scan    # ← 如果 wifi 未激活，scan 在 utils 中被消除
scan()                    # ← ImportError 或 NameError
```

**解决**：在 main.py 中也用 `with feature("wifi")` 包裹调用点。

### 陷阱 4：标签大小写

- target 自动转大写：`--target esp32` → `ESP32`
- feature 保持原样：`--feature SdCard` 需要代码中用 `feature("SdCard")`

### 陷阱 5：双重依赖过滤不匹配

manifest.py 用 `features` 过滤，代码内部也有 `with feature()`。如果 manifest 没过滤但代码内全部消除，文件刷入一个空文件，浪费设备空间。建议要么在 manifest 层过滤，要么在代码层 guard，不要双重依赖。

## 10. 总结

| 场景 | 推荐工具 | 效果 |
|------|----------|------|
| 多平台同源 | `@target("...")` 装饰器 | 同名函数按平台保留实现 |
| 可选功能裁剪 | `with feature("...")` 块 | 按 tag 保留或消除代码块 |
| 开发/生产差异 | `feature("debug")` / `feature("prod")` | 一行命令切换构建模式 |
| 文件级裁剪 | manifest.py `features` 参数 | 不匹配的文件不刷入 |
| 平台抽象层 | `@target` + `with target` 合并到 PAL 文件 | 业务代码零平台耦合 |

条件编译的核心价值：**一份源码，多套构建，按需组合**。
